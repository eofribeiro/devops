\subsection*{O que é isso?!}
\textbf{HiberNatural} é uma biblioteca em Java que faz o mapeamento "objeto-texto" de chamadas ao \textbf{Mainframe} via \textbf{NatAPI}. Pela sua funcionalidade e característica de camada, faz o papel do \textbf{Hibernate} para o \textbf{Natural}. Nesta perspectiva, a \textbf{NatAPI} seria o análogo ao \textbf{Driver JDBC}.

Com ela, há maior transparência nos processos de acesso ao \textbf{Natural}, e o programador não precisa se preocupar com tratamento de \textit{substrings}, aumentando a produtividade e reduzindo o risco de erros. Basta criar um objeto, executar um método e receber outro objeto!

\subsection*{Arquitetura}
\includegraphics{esquema-hn.png}

\begin{enumerate}
\item objeto do tipo \textbf{DTO} é instanciado e tem suas propriedades atribuídas
\item HiberNatural é executado
  \begin{itemize}
  \item a partir do objeto DTO, cria-se a string de envio ao programa Natural
  \item chamada ao Mainframe através da NatAPI
    \begin{itemize}
    \item leitura de RC (\textit{return code}), MSG (mensagem de erro) e da saída (\textit{data parameter})
    \end{itemize}
  \item parse da string de retorno e criação do objeto de saída
  \end{itemize}
\item objeto do tipo \textbf{POJO} é criado e retornado
\end{enumerate}

\subsection*{Exemplo de Código}
\subsubsection*{NatAPI Pura}
\begin{verbatim}
// manipulação da string de envio
String entrada = "";
//...

// chamada ao mainframe via NatAPI
Natural nat = new Natural("PRG001", entrada);

// verificar código de retorno
if (nat.getRC() == 0) {
	
	// interpretação da string de retorno
	String saida = nat.getDataParameter();
	
	// manipulação da string de retorno
	//...
	
	// criação de objetos e atribuição de valores
	//...

// se houve erro...
} else {

	// captura da mensagem de erro
	String msg = nat.getMsg();
	
	// tratamento de erro
	//...
}
\end{verbatim}

\subsubsection*{HiberNatural}
\begin{verbatim}
// criação do objeto de envio (DTO)
SITVN041DTO dto = new SITVN041DTO();
dto.setPlaca("ASA0610");
dto.setNumProcesso(2006001300041015L);

// chamada ao HiberNatural e criação do objeto de retorno (POJO)
NaturalSITVN041DAO dao = new NaturalSITVN041DAO();
SITVN041 pojo = (SITVN041) dao.executar(dto);
\end{verbatim}

\subsection*{Instalação}
As seguintes bibliotecas (versão igual ou superior) são necessárias:

\begin{enumerate}
\item \textbf{Jasppion (NatAPI)}: jasppion-1.0.7.jar
\item \textbf{Log4j}: log4j-1.2.8.jar
\item \textbf{Commons HTTP Client}: commons-httpclient-2.0.jar
\item \textbf{Commons Languages}: commons-lang-1.0.1.jar
\item \textbf{Commons Logging}: commons-logging-1.0.4.jar
\end{enumerate}

Para realizar a instalação, basta:

\begin{itemize}
\item copiar os arquivos acima e a \textbf{hibernatural-$<$versão$>$.jar} para o diretório \textit{lib} do aplicativo
\item criar o arquivo de mapeamento \textbf{natapi.xml} no diretório \textit{src}
\item criar o arquivo de usuários e senhas \textbf{PRG.properties} (onde \textit{PRG} é a sigla do aplicativo no Natural) no diretório \textit{src}
\item criar classe do tipo DAO, herdada da \textbf{BaseNaturalDAO}
\end{itemize}

\subsection*{Mapeamento}
Para a execução do \textbf{HiberNatural}, é preciso criar o arquivo \textbf{natapi.xml}, o qual deve ficar num diretório definido no \textit{CLASSPATH} do aplicativo (geralmente \textit{src}). Este \textbf{XML} obedece ao arquivo \textbf{DTD} \textbf{natapi.dtd}, incluído no arquivo \textbf{JAR}, que contém todas as regras para sua validação.

Não pode existir mais de um \textbf{programa} com o mesmo nome, e cada um deles deve ter \textbf{entrada} e \textbf{saída}. Nestas últimas tags estarão contidos um ou mais atributos (\textbf{constantes}, \textbf{campos}, \textbf{listas} e \textbf{objetos}).

Como a biblioteca é baseada em programação orientada a objetos, é um pré-requisito que as classes especificadas estejam criadas (e já compiladas) e pelo menos com as propriedades indicadas. Há também sensitividade de caixa no arquivo XML.

\subsubsection*{Validação}
Para verificar se o arquivo de configurações \textbf{"natapi.xml"} contém erros, faça o seguinte:

\begin{enumerate}
\item inclua no "natapi.xml" a indicação da \textbf{DTD}, como no exemplo abaixo:
\begin{verbatim}
<?xml version="1.0" encoding="ISO-8859-1"?>

<!DOCTYPE natapi-defs PUBLIC "natapi" "http://10.15.61.6/hibernatural/natapi-2.1.dtd">

<natapi-defs>
	...
</natapi-defs>
\end{verbatim}

\item no \textbf{Eclipse}, abra o arquivo "natapi.xml", clique com o botão direito e selecione a opção \textbf{Validate}
\end{enumerate}

\subsection*{Depuração}
As classes que compõem o Hibernatural fazem uso da biblioteca \textbf{Log4j} para geração de LOGs. Um \textit{appender} chamado \textbf{"gov.pr.celepar.util.natapi"} é instanciado internamente, de modo que o arquivo \textbf{log4j.properties} pode ser configurado também para chamadas ao Mainframe. Veja exemplo:

\begin{verbatim}
log4j.rootLogger=DEBUG, stdout
log4j.logger.gov.pr.celepar.util.natapi=DEBUG, arquivo

# exibição no console
log4j.appender.stdout=org.apache.log4j.ConsoleAppender
log4j.appender.stdout.Target=System.out
log4j.appender.stdout.layout=org.apache.log4j.PatternLayout
log4j.appender.stdout.layout.ConversionPattern=%m%n
#log4j.appender.stdout.Threshold=INFO
log4j.appender.stdout.Threshold=DEBUG

# inclusão de linhas no arquivo
log4j.appender.arquivo=org.apache.log4j.RollingFileAppender
log4j.appender.arquivo.File=/tmp/natapi.log
log4j.appender.arquivo.MaxFileSize=500KB
log4j.appender.arquivo.MaxBackupIndex=3
log4j.appender.arquivo.layout=org.apache.log4j.PatternLayout
log4j.appender.arquivo.layout.ConversionPattern=%m%n
log4j.appender.arquivo.Threshold=DEBUG

\subsection*{CHANGELOG}
\subsubsection*{Versão 2.6}
\paragraph{}\textbf{Correção de bugs}\\

Erro \textit{NoSuchMethodException} era levantado quando diversas chamadas ao HiberNatural eram feitas de modo simultâneo. Isso foi corrigido com a implementação de um método \textbf{synchronized} numa abordagem do tipo \textbf{singleton}.

\paragraph{}\textbf{Melhorias na leitura das definições}\\

Anteriormente, a cada chamada de programa, o arquivo de configurações (natapi.xml) era relido. O código que fazia a leitura das definições de entrada e saída para um programa foi modificado, de modo que a leitura é realizada uma única vez, na primeira chamada de qualquer programa.

Com isso, as execuções via HiberNatural tendem a ser muito mais rápidas.

\subsubsection*{Versão 2.5}
\paragraph{}\textbf{Melhorias no tratamento de exceções}\\

Serão validados, ao instanciar a classe da HiberNatural:

\begin{itemize}
\item a existência do arquivo de propriedades \textbf{natapi.xml}
\item a existência do programa especificado no arquivo
\item a existência de parâmetros de entrada e saída para o programa
\end{itemize}

\subsubsection*{Versão 2.4}
\paragraph{}\textbf{Correção de bugs}\\

\begin{itemize}
\item corrigido bug na criação de listas com separador
\item datas inválidas vindas do Mainframe agora são desconsideradas (valor \textit{null})
\end{itemize}

\paragraph{}\textbf{Aviso sobre o tamanho de campos}\\

Alteração para que a biblioteca passe a lançar uma exceção nos casos onde o valor passado para um campo seja de tamanho maior que o tamanho definido nas propriedades. Por exemplo, nesta definição de campo:

\begin{verbatim}
<campo nome="descrProcedencia" tipo="String" tam="3"/>
\end{verbatim}
Se no atributo \textit{descrProcedencia} o tamanho da String for maior que 3, lançar uma exceção dizendo que há um problema no tamanho deste campo.

Essa melhoria ajudará na identificação de problemas dessa natureza em programas com muitos parâmetros de entrada ou saída.

\subsubsection*{Versão 2.3}
\paragraph{}\textbf{Classe de exceção}\\

Foi criada a classe \textit{BaseNaturalException}, que será lançada no lugar da \textit{ApplicationException} quando houver erro da \textbf{NatAPI}, ou seja, RC (\textit{return code}) diferente de zero.

\paragraph{}\textbf{Flexibilidade}\\

Agora o \textbf{HiberNatural} não depende mais da biblioteca \textbf{framework.jar}. As exeções lançadas serão exclusivamente do tipo \textit{Exception}, e não mais \textit{ApplicationException}. Com isso, o \textbf{HiberNatural} torna-se mais flexível, podendo ser utilizado em projetos sem o padrão \textit{Framework}.

\paragraph{}\textbf{Novos tipos}\\

Foi adicionado suporte aos seguintes tipos de dados:

\begin{itemize}
\item \textbf{Float}: apesar de ter a metade da capacidade do \textit{Double}, não gera erros de arredondamento em certos casos
\item \textbf{BigDecimal}: numérico com grande capacidade de armazenamento, comum em classes geradas pelo \textit{Middlegen} para o \textit{Hibernate}
\end{itemize}

Para o tipo \textbf{BigDecimal}, foi utilizado o construtor por \textit{String}, devido à imprecisão apresentada com o tipo \textit{double}.

\paragraph{}\textbf{Melhoria na depuração}\\

Exibição de maneira mais explícita, na saída de erro, tentativas sem sucesso de criação de classes (\textit{ClassNotFoundException}).

\subsubsection*{Versão 2.1}
\paragraph{}\textbf{Constantes}\\

Esta funcionalidade faz com que trechos da string de retorno sejam automaticamente ignorados e dispensa a criação de propriedades "fixas" em objetos DTO para a string de envio.

\begin{itemize}
\item \textbf{$<$aspas/$>$}: representa "
\item \textbf{$<$virgula/$>$}: representa ,
\item \textbf{$<$espaco tam="5"/$>$}: representa uma cadeia de espaços do tamanho especificado
\item \textbf{$<$fixo valor="XXX"/$>$}: inclui a constante especificada
\end{itemize}

\paragraph{}\textbf{Tag Objeto}\\

Com isso, não será necessário usar mais a tag \textbf{$<$lista$>$} contendo apenas 1 item (\textit{qtd="1"}). Um objeto inteiro pode ser lido (para a entrada) ou criado (para a saída). Exemplo:

\begin{verbatim}
<objeto nome="sitvn200R1" classe="gov.pr.detran.veiculos.pojo.SITVN200R1">
	<campo nome="tipoRegistro" tipo="Byte" tam="1"/>
\end{verbatim}

\paragraph{}\textbf{Conceito de contêiner}\\

As tags \textbf{$<$lista$>$} e \textbf{$<$objeto$>$} implementam esta interface, e por isso são capazes de armazenar outros atributos. Assim, podem ser feitas listas dentro de listas, por exemplo. Recursividade é possível! Exemplo:

\begin{verbatim}
<lista nome="registros" classe="gov.pr.detran.veiculos.pojo.SITVN601" qtd="$qtde">
	<lista nome="motivosCombinados" classe="gov.pr.detran.veiculos.pojo.MotivoCombinado" qtd="6" separador="">
		<campo nome="codMotivoCombinado" tipo="Byte" tam="2"/>
\end{verbatim}

\paragraph{}\textbf{Separador e Extremos}\\

Em alguns casos, os programas \textbf{Natural} esperam a string de entrada com os campos separados por um determinado caracter. Anteriormente, era preciso criar campos fixos do tipo \textit{String} entre os campos reais. Agora isso não será mais preciso, e os códigos ficarão muito mais enxutos e legíveis.
Contêineres (Listas e Objetos) podem ter separadores diferenciados.
Se a opção \textit{extremos} for especificada, a string será limitada por estes caracteres, como aspas duplas ou vírgulas. Exemplos:

\begin{verbatim}
<entrada extremos="&#034;" classe="gov.pr.detran.habilitacao.dto.DutHN151DTO">
<entrada separador="," extremos="," classe="gov.pr.detran.veiculos.dto.SITVN601DTO">
	<lista nome="funcoes" classe="gov.pr.detran.veiculos.pojo.FuncaoEspecial" qtd="10" separador="">
\end{verbatim}

\paragraph{}\textbf{Listas de tamanho variável}\\

Com esse recurso, o tratamento de \textit{RecordSets} do \textbf{Natural} pode ser feito sem dificuldades! O caso dos primeiros 5 bytes que a \textbf{NatAPI} retorna ("00001" na maioria dos casos) torna-se um caso particular desta função. Para isso, a opção \textit{qtd} da lista, que geralmente contém um número fixo, deve ser preenchida com uma expressão. Para utilizar campos da própria classe, emprega-se o sinal de cifrão, semelhantemente à linguagem PHP. Exemplo:

\begin{verbatim}
<campo nome="qtdRegistros" tipo="Integer" tam="5"/>
<lista nome="registros" qtd="$qtdRegistros">
\end{verbatim}

\paragraph{}\textbf{Redefinição condicional}\\

Com isso, a classe do objeto a ser criado e populado é definida em tempo de execução! Na expressão da condição, podem ser usados valores de campos já atribuídos na classe anterior, utilizando o sinal de cifrão.
No exemplo abaixo, as classes \textit{SITVN601R1} e \textit{SITVN601R3} estendem a \textit{SITVN601}, e a decisão sobre qual delas criar fica a critério da análise da expressão lógica. Existe ainda, através do \textit{método de redefinição}, a opção de copiar atributos comuns da classe herdada, a fim de deixar o código mais limpo.

\begin{verbatim}
<lista nome="registros" classe="gov.pr.detran.veiculos.pojo.SITVN601" qtd="$qtde">
	<campo nome="evento" tipo="Integer" tam="4"/>
	<campo nome="codTransacao" tipo="Integer" tam="3"/>
	<campo nome="tipoRegistro" tipo="Byte" tam="1"/>
	<redefinicao metodo="copiar-atributos" classe="gov.pr.detran.veiculos.pojo.SITVN601R1" condicao="$tipoRegistro == 1">
		<campo nome="placa" tipo="String" tam="7"/>
		<campo nome="renavam" tipo="Long" tam="9"/>
		<campo nome="tipoVeiculo" tipo="Short" tam="2"/>
	</redefinicao>
	<redefinicao metodo="copiar-atributos" classe="gov.pr.detran.veiculos.pojo.SITVN601R3" condicao="$tipoRegistro == 3">
		<campo nome="numProtocolo" tipo="Long" tam="16"/>
		<campo nome="codMotivoPrincipal" tipo="Short" tam="2"/>
	</redefinicao>
\end{verbatim}
